# Daily Briefing — JSON Schema & Rendering Specification

> **Authoritative shape contract: `briefing_schema.json`** (co-located with this file). The Phase 4.5 validation gate in the `daily-dashboard` skill enforces structural shape, types, enums, required fields, and date/datetime formats against that JSON Schema automatically. This markdown is the human-readable companion — narrative, rendering guidance, and the merge/ordering rules the validator can't check on its own.

This document defines the canonical JSON structure for daily briefings generated by the `day-starter` skill and any intraday update passes. It is the source of truth for both the generation side (what Claude produces) and the display side (how a viewer component renders it).

The schema is designed to be updated multiple times per day (morning, midday, evening). Each update replaces the full JSON for that day. A display component reads the JSON and renders a live view; the Markdown files these replaced were static snapshots.

Configuration that does not change day-to-day (channel lists, deal registry, intelligence categories, email filter rules) lives in `briefing_config.yaml` alongside this file.

---

## Design Principles

1. **No derived fields.** Do not store counts that can be computed from arrays. Do not store a "critical email count" when `email.act_today` is an array whose length is the count.
2. **Source references over duplication.** Action items are a synthesis across email, Slack, and calendar. They reference their source by id — they do not repeat the full source content.
3. **One canonical id per entity.** Deals, channels, intelligence categories, and initiatives use stable ids defined in `briefing_config.yaml`. These ids appear as `deal_tag`, `category_id`, and `initiative` throughout the JSON, enabling a display component to enrich, group, and filter items without the briefing JSON needing to duplicate labels or metadata.
4. **Nullable over absent.** Use `null` rather than omitting optional fields. This makes the schema stable across updates and easier to render without null-guards.
5. **ISO 8601 for all datetimes.** All timestamps are ISO 8601 with timezone offset. Human-readable time strings (`"15:45"`) are allowed only for display-facing fields like `calendar.events[].start`.

## Top-Level Structure

```json
{
  "meta":          { ... },
  "greeting":      { ... },
  "meeting_preps": [ ... ],
  "calendar":      { ... },
  "email":         { ... },
  "slack":         { ... },
  "intelligence":  [ ... ],
  "action_items":  [ ... ],
  "focus_prompt":  "string",
  "fun":           { ... }
}
```

## Section: `meta`

Administrative metadata about this briefing document.

```json
"meta": {
  "briefing_date":        "2026-04-27",
  "generated_at":         "2026-04-27T08:15:00+02:00",
  "timezone":             "Europe/Berlin",
  "update_sequence":      1,
  "last_successful_run":  null,
  "run_type":             "full",
  "next_meeting": {
    "title":       "ABM LLMObs Sync",
    "starts_at":   "2026-04-27T09:00:00+02:00",
    "minutes_away": 45
  }
}
```

| Field | Type | Description |
|---|---|---|
| `briefing_date` | `YYYY-MM-DD` | Calendar date this briefing covers. Used as the partition key for a day's updates. |
| `generated_at` | ISO 8601 datetime | When this version of the JSON was produced. Displayed as "Briefing generated at HH:MM TZ" in the footer. |
| `timezone` | IANA string | The user's local timezone for this briefing. Sourced from `briefing_config.yaml`. |
| `update_sequence` | integer ≥ 1 | 1 = morning briefing, 2 = first intraday update, etc. Display can show "Updated at HH:MM" for sequence > 1. |
| `last_successful_run` | ISO 8601 datetime \| null | The `generated_at` timestamp of the **previous** successfully completed run. Set to `null` on the first run of the day. The `daily-dashboard` skill uses this to determine how far back to look for new Slack messages and email. Updated by the skill immediately after a successful write. |
| `run_type` | enum | `"full"` \| `"midday"` \| `"incremental"`. Which generation pass produced this version. `full` = first run of the day (all sections, full intel); `midday` = pass within ±90 min of a scheduled `intraday_updates` time (all comm channels + abbreviated intel); `incremental` = any other intraday pass (comm channels only, no web intel). |
| `next_meeting` | object \| null | The next calendar event coming up. Shown in the footer countdown. Null if no meetings remain today. |
| `next_meeting.title` | string | Short event title. |
| `next_meeting.starts_at` | ISO 8601 datetime | Start time of the next meeting. |
| `next_meeting.minutes_away` | integer | Minutes from `generated_at` to `starts_at`. Recomputed on each update pass. |

## Section: `greeting`

A short personalized header for the day.

```json
"greeting": {
  "text":         "Good morning Stefan — Monday, April 27",
  "context_note": "Travel day: LH401 departs JFK 15:40 ET → FRA. Check in via Navan (BALB4P)."
}
```

| Field | Type | Description |
|---|---|---|
| `text` | string | Greeting line. Format: `"Good morning [name] — [Weekday], [Month Day]"`. Intraday updates use `"Midday update"` or `"Evening update"`. |
| `context_note` | string \| null | A single high-priority framing note that colours the whole day — travel, a major deadline, an unusual schedule. Not for routine items. Null on normal days. |

**Rendering:** Display as a full-width header banner. The `context_note`, if present, renders as a highlighted callout block immediately below the banner.

## Section: `meeting_preps`

Prepared meeting brief files for today's external meetings.

```json
"meeting_preps": [
  {
    "time":  "15:45",
    "title": "[Aspen] Daily Standup · Julien Launay (Adaptive ML)",
    "file":  "meeting-prep-1545-adaptive-ml-aspen.md",
    "deal_tag": "aspen"
  }
]
```

| Field | Type | Description |
|---|---|---|
| `time` | `"HH:MM"` | Meeting start time in local timezone. Used for sorting and display. |
| `title` | string | Human-readable meeting title, matching the calendar event title. |
| `file` | string \| null | Filename of the prep document, relative to the day's briefing folder (e.g. `briefings/2026_04_27/`). The display component should link this as a `computer://` path. |
| `deal_tag` | string \| null | Deal id from `briefing_config.yaml`. Null for prep docs not tied to a specific deal. |
| `meeting_id` | string \| null | Producer-side calendar identifier (e.g. `cal_4`) — links this prep back to the corresponding `calendar.events[].id`. Optional. |
| `time_iso` | ISO 8601 datetime \| null | Full ISO timestamp of the meeting start (mirror of `time` for sorting/comparison). Optional. |
| `type` | enum \| null | Same enum as `calendar.events[].type` (`external | internal | …`). Optional. |

**Rendering:** Render as a compact table. Each row: time + title + linked file icon. If `deal_tag` is set, apply the deal's accent color to the row.

## Section: `calendar`

Today's calendar events and any scheduling conflicts.

```json
"calendar": {
  "summary": "13 events · Heavy back-end loaded day — free until 11:00, then solid from 14:00 onward",
  "events": [ ... ],
  "conflicts": [ ... ]
}
```

### `calendar.summary`

A one-line human characterization of the day's shape. Written by the generation pass — not derived mechanically. Examples:
- `"10 events · Heavy demo day · Travel tonight"`
- `"Light day — 2 external meetings, free morning"`
- `"14 events — packed; late-night Aspen sessions"`

### `calendar.events[]`

```json
{
  "start":        "15:45",
  "end":          "16:00",
  "title":        "[Aspen] Daily Standup",
  "participants": ["Julien Launay (adaptive-ml.com)", "Joe Bennett", "Asher Kinyon"],
  "type":         "external",
  "is_external":  true,
  "urgency":      "high",
  "deal_tag":     "aspen",
  "initiative":   "aspen_merger_agreement",
  "links": {
    "doc":   null,
    "zoom":  "https://datadog.zoom.us/j/95291761345",
    "other": null
  },
  "notes": "Late-stage M&A · prep file ready"
}
```

| Field | Type | Description |
|---|---|---|
| `start` | `"HH:MM"` | Start time in local timezone. |
| `end` | `"HH:MM"` | End time in local timezone. |
| `title` | string | Event title as it appears in calendar. |
| `participants` | string[] | List of participant names or email addresses. Omit Stefan himself — implied. |
| `type` | enum | `"external"` \| `"internal"` \| `"personal_block"` \| `"personal"` \| `"declined"`. Determined by classification rules in `briefing_config.yaml`. |
| `is_external` | boolean | True if any participant matches an external domain from `calendar_classification.external_indicator_domains`. Redundant with `type === "external"` but kept for easy filtering. |
| `urgency` | enum | `"critical"` \| `"high"` \| `"medium"` \| `"low"`. How important this meeting is to attend and prepare for. `critical` = deal-blocking or exec-attended; `high` = external or significant internal; `medium` = routine internal; `low` = optional / informational. Personal blocks are always `"low"`. |
| `deal_tag` | string \| null | Deal this event belongs to, if any. |
| `initiative` | string \| null | Initiative `id` from `briefing_config.yaml initiatives[]`. Assigned by keyword-matching the event title, description, and participants against each initiative's `keywords` list — first match wins. Null if no initiative matches. **Never use free-text here** — if a new initiative is needed, add it to config first so the id is stable for grouping and filtering. Can coexist with `deal_tag` (e.g. a meeting can be `deal_tag: "aspen"` + `initiative: "aspen_merger_agreement"` to identify which thread within the deal). |
| `links.doc` | string \| null | Google Doc, Notion, or other reference document for this meeting. |
| `links.zoom` | string \| null | Video call link. |
| `links.other` | string \| null | Any other relevant link (calendar invite, spreadsheet, etc.). |
| `notes` | string \| null | Short qualifier: deal stage, who's present (especially executives), priority signal, or prep status. |
| `id` | string \| null | Producer-side stable event identifier (e.g. `cal_4`). Optional. |
| `start_iso` | ISO 8601 \| null | Full timestamp of `start` (mirror with timezone). Optional. |
| `end_iso` | ISO 8601 \| null | Full timestamp of `end`. Optional. |
| `category` | enum \| null | Same enum as `type`. Currently a duplicate of `type`; producer emits both. Optional. |
| `rsvp_status` | string \| null | Stefan's RSVP state on the invite (`accepted`, `tentative`, `declined`, `needsAction`). Optional. |
| `location` | string \| null | Physical location or room. Optional. |
| `action` | string \| null | Per-event action item if any (separate from the cross-source `action_items[]` synthesis). Optional. |

**Rendering:** Render as a timeline table. Rows are sorted by `start`. Apply visual weight by `type` and `urgency`:
- `external` + `urgency: critical/high` → bold title, red left border
- `internal` → normal weight, green left border
- `personal_block` → muted/grey, no border
- `declined` → strikethrough
If `deal_tag` is set, add a small deal badge. If `initiative` is set and differs from `deal_tag`, show it as a secondary label. Show `links` as icon buttons (📄 doc, 📹 zoom). Show `notes` as a small tag or tooltip.

### `calendar.conflicts[]`

```json
{
  "time":         "18:00",
  "event_titles": ["Push Security <> Datadog", "Erhan/Stefan × Marc/Matt Follow-up"],
  "description":  "Push Security (18:00–19:00) overlaps entirely with Erhan/Stefan × Marc/Matt (18:00–18:30). Stefan's status on Push invite is needsAction.",
  "action_needed": "Accept or flag conflict to Gabriel/Erhan before standup."
}
```

| Field | Type | Description |
|---|---|---|
| `time` | `"HH:MM"` | Time at which the conflict starts. |
| `event_titles` | string[] | The titles of the conflicting events (2 or more). |
| `description` | string | Full explanation of the overlap and any relevant context (invite status, who is affected). |
| `action_needed` | string \| null | Specific resolution action, if one is clear. Null if the conflict is informational only. |

**Rendering:** Render as a warning callout block, below the events table. Each conflict gets a `⚠️` icon and the `description` text. If `action_needed` is set, show it in bold as a sub-item.

## Section: `email`

Curated email items organized by required action level.

```json
"email": {
  "act_today": [ ... ],
  "fyi":       [ ... ],
  "no_action_summary": "Datadog Daily Digests (5), LinkedIn invitations (10+), Seeking Alpha, Navan survey, Confluence digest — filed."
}
```

### `email.act_today[]`

Items requiring a response, decision, or concrete action today.

```json
{
  "id":         "email_act_1",
  "from":       "Oli Pomel",
  "subject":    "Re: Aspen gitstat analysis",
  "received":   "2026-04-27T02:33:00+02:00",
  "url":        "https://mail.google.com/mail/u/0/#all/19dbbcf416d50d7c",
  "summary":    "Oli replied to Stefan's gitstat analysis asking for a Claude stack-rank of committers by quantity, impact, and complexity.",
  "action":     "Reply with status / run the Claude committer analysis before the 15:45 standup.",
  "urgency":    "critical",
  "deal_tag":   "aspen",
  "initiative": "aspen_eng_diligence",
  "tags":       ["aspen", "diligence", "oli"]
}
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique id within this briefing, used by `action_items[].source_ref`. Format: `"email_act_N"`. |
| `from` | string | Sender display name. |
| `subject` | string | Email subject line. |
| `received` | ISO 8601 datetime | When the email arrived. |
| `url` | string | Direct Gmail deep-link to the thread. |
| `summary` | string | 1–3 sentence summary of the email content. Should be specific enough to act without opening the email. |
| `action` | string | Exactly what needs to happen. Starts with a verb. Includes deadline if time-sensitive. |
| `urgency` | enum | `"critical"` \| `"high"` \| `"medium"` \| `"low"`. `critical` = blocks a meeting, decision, or deal step happening today; `high` = time-sensitive reply needed today; `medium` = should happen today but flexible; `low` = do when possible. All `act_today` items are at least `"medium"` — use `"critical"` and `"high"` to distinguish within the section. |
| `deal_tag` | string \| null | Deal id if this email belongs to an active deal. |
| `initiative` | string \| null | Initiative `id` from `briefing_config.yaml initiatives[]`. Auto-assigned by matching the email subject and body against each initiative's `keywords` list — first match wins. Null if no initiative matches. **Always use the config id, never free text** — consistent ids are required for grouping and filtering across email, Slack, and calendar. |
| `tags` | string[] | Free-form keywords for grouping/filtering. Include deal name, sender slug, topic area. |

### `email.fyi[]`

Items worth awareness — relevant context, pre-reads, lightweight follow-ups — but no hard action today.

```json
{
  "id":         "email_fyi_1",
  "from":       "Ron Feldman",
  "subject":    "Culminate 3/6/9 & Bits AI Security Analyst Exec Update",
  "received":   "2026-04-27T01:41:00+02:00",
  "url":        "https://mail.google.com/mail/u/0/#all/19dcc99d2729fc4b",
  "summary":    "Pre-read for tonight's 19:00 exec update on Bits AI Security Analyst and Culminate. Oli is attending.",
  "context":    "Pre-read for 19:00 exec update — read before the meeting.",
  "urgency":    "medium",
  "deal_tag":   null,
  "initiative": "bits_ai_security",
  "tags":       ["bits", "culminate", "exec-update", "oli"]
}
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique id within this briefing. Format: `"email_fyi_N"`. |
| `from` | string | Sender display name. |
| `subject` | string | Email subject line. |
| `received` | ISO 8601 datetime | Receive time. |
| `url` | string \| null | Gmail deep-link. Null if URL is unavailable. |
| `summary` | string | What the email contains. |
| `context` | string \| null | Why it's in FYI and what to do with it — e.g. `"Pre-read for 17:00 meeting"`, `"FYI on deal status"`, `"Monitor for outcome"`. |
| `urgency` | enum | `"critical"` \| `"high"` \| `"medium"` \| `"low"`. For FYI items, typically `"medium"` (worth reading today) or `"low"` (background awareness). Use `"high"` for FYI items tied to a meeting happening soon. |
| `deal_tag` | string \| null | Deal id if applicable. |
| `initiative` | string \| null | Initiative `id` from config — same matching rules as `act_today`. Especially useful for FYI items that belong to an ongoing internal initiative without a deal tag, e.g. `"bits_ai_security"`, `"culminate_369"`, `"q1_bod"`. |
| `tags` | string[] | Keywords. |

### `email.no_action_summary`

A single string listing the categories of auto-filed email with approximate counts. This is intentionally low-resolution — just enough to confirm nothing was missed without cluttering the view.

Example: `"Datadog Daily Digests (5), LinkedIn invitations (12), Seeking Alpha (2), Navan survey, Confluence digest — all filed."`

**Rendering for the email section:** Render `act_today` as prominent cards with a red/orange left border. Each card shows: from + subject + received time + summary + action (bolded). `fyi` items render as a lighter list with grey border. `no_action_summary` renders as a collapsed or dimmed single line at the bottom.

## Section: `slack`

Curated Slack activity across VIP channels and relevant DMs.

```json
"slack": {
  "since": "Friday EOD",
  "channels": [ ... ],
  "dms": [ ... ]
}
```

### `slack.since`

Human-readable lookback description: `"yesterday EOD"`, `"Friday EOD"`, `"2 hours ago"`. Not a parseable datetime — it's a display label.

### `slack.channels[]`

```json
{
  "channel_id":     "C0AQHD40GTH",
  "channel_name":   "#int-aspen-corpdev",
  "url":            "https://dd.slack.com/archives/C0AQHD40GTH",
  "deal_tag":       "aspen",
  "activity_level": "high",
  "messages": [
    {
      "author":    "Brian Peters",
      "timestamp": "2026-04-26T18:23:00+02:00",
      "summary":   "Posted the latest Merger Agreement Issues List with Special Holdback language carving out known security/infra issues.",
      "links": [
        { "label": "Merger Agreement Issues List", "url": "https://docs.google.com/document/d/..." },
        { "label": "Security Onboarding Plan",     "url": "https://docs.google.com/document/d/..." }
      ],
      "action": null
    }
  ]
}
```

| Field | Type | Description |
|---|---|---|
| `channel_id` | string | Slack channel ID. Must match an entry in `briefing_config.yaml vip_slack_channels`. |
| `channel_name` | string | Display name with `#` prefix. |
| `url` | string | Direct link to the Slack channel. |
| `deal_tag` | string \| null | Deal this channel belongs to. |
| `activity_level` | enum | `"high"` \| `"medium"` \| `"low"` \| `"quiet"`. Qualitative signal. `"quiet"` = nothing new since last briefing. |
| `messages[].author` | string | Slack display name of the poster. |
| `messages[].timestamp` | ISO 8601 \| null | When the message was sent. |
| `messages[].summary` | string | 1–3 sentence description of the message content. |
| `messages[].links[]` | array | Named links extracted from the message (Google Docs, spreadsheets, etc.). |
| `messages[].action` | string \| null | If this specific message requires a follow-up action, describe it. Otherwise null. |
| `messages[].url` | string \| null | Direct deep-link to the specific message (`https://dd.slack.com/archives/{channelId}/p{ts_no_dot}`). Optional. |

Channels with `activity_level: "quiet"` should still appear in the digest with a single `"No new messages since [since]."` entry so the user has confirmation coverage.

### `slack.dms[]`

Direct messages and group DMs relevant to include.

```json
{
  "with":    "Gabriel Levy",
  "url":     "https://dd.enterprise.slack.com/archives/D053QL8MQ3C/p1777031411462369",
  "summary": "Gabriel asked if you're still free this morning — you confirmed yes. Loop closed.",
  "action":  null
}
```

| Field | Type | Description |
|---|---|---|
| `with` | string | Name(s) of the other participant(s). |
| `url` | string | Deep link to the DM or specific message. |
| `summary` | string | What was said and the current state. |
| `action` | string \| null | Follow-up needed, if any. |

**Rendering for slack section:** Render each channel as a collapsible card. Header shows channel name + activity level badge + deal tag. Messages render as a compact thread-style list with author + summary + links. DMs appear as a separate section below channels.

## Section: `intelligence`

Market intelligence, tech news, and competitive signals, organized by category.

```json
"intelligence": [
  {
    "category_id": "ai_llms",
    "items": [
      {
        "headline":   "GPT-5.5 shipped (Apr 23)",
        "detail":     "OpenAI's omnimodal model hits 88.7% SWE-Bench Verified. Inference pricing down ~50% vs Jan 2026, compressing margins across the stack.",
        "date":       "2026-04-23",
        "source":     "Fazm Blog",
        "url":        "https://fazm.ai/blog/llm-news-april-2026",
        "tags":       ["openai", "gpt-5", "pricing"],
        "relevance":  "Inference cost pressure directly relevant to Adaptive ML's positioning and Aspen deal thesis."
      }
    ]
  }
]
```

The `intelligence` array is ordered by `category_id` matching the `order` field in `briefing_config.yaml intelligence_categories`.

### `intelligence[].category_id`

Must match one of the `id` values in `briefing_config.yaml intelligence_categories`. The display component looks up the label and icon from config — do not duplicate them in the JSON.

### `intelligence[].items[]`

| Field | Type | Description |
|---|---|---|
| `headline` | string | Short, punchy title. May include date in parentheses if useful for scanning. |
| `detail` | string | 2–4 sentences of elaboration. Should include the "so what" — why this matters, not just what happened. |
| `date` | `YYYY-MM-DD` \| null | Publication or event date. Null if not determinable. |
| `source` | string | Publication name (e.g. `"TechCrunch"`, `"Crunchbase"`, `"GlobeNewswire"`). |
| `url` | string \| null | Source URL. Null if paywalled or not linkable. |
| `tags` | string[] | Keywords: company names, topics, product names. |
| `relevance` | string \| null | 1-sentence explicit connection to Stefan's current work, deals, or Datadog. Include this whenever the connection is non-obvious. Null for items where relevance is self-evident (e.g. Datadog earnings in the Datadog category). |

**Rendering:** Render each category as a collapsible section with the category icon and label as the header. Items render as a bulleted list: `headline` in bold, followed by `detail` inline, followed by a linked `source` citation. If `relevance` is set, show it as an indented italic note beneath the item.

## Section: `action_items`

A synthesized, prioritized list of concrete things Stefan needs to do today. This is a cross-source consolidation — it draws from email, Slack, and calendar. Each item has a `fingerprint` that is stable across runs, enabling completion state to survive full regeneration. `source_ref` links back to the originating item so the display component can show context without duplicating content.

```json
"action_items": [
  {
    "id":           "action_1",
    "fingerprint":  "email-19dbbcf416d50d7c",
    "text":         "Reply to Oli on Aspen gitstat committer stack-rank — run Claude analysis before 15:45 standup.",
    "priority":     "high",
    "deadline":     "before 15:45",
    "source_type":  "email",
    "source_ref":   "email_act_1",
    "url":          "https://mail.google.com/mail/u/0/#all/19dbbcf416d50d7c",
    "deal_tag":     "aspen",
    "done":         false,
    "created_at":   "2026-05-01T08:14:00+02:00",
    "completed_at": null
  }
]
```

| Field | Type | Description |
|---|---|---|
| `id` | string | Positional id within this run's output. Format: `"action_N"` for newly derived items; `"action_done_N"` for items preserved from the base via fingerprint match (so done items remain identifiable in run summaries even after sorting). Used only for display ordering — **do not use for deduplication across runs**. |
| `fingerprint` | string | **Stable cross-run identity key.** Derived deterministically from the underlying source so the same real-world action always produces the same fingerprint regardless of run order or list position. Construction rules — see below. |
| `text` | string | Full action statement. Starts with an imperative verb. Includes enough context to act without looking up the source. May include deadline inline. |
| `priority` | enum | `"high"` \| `"medium"` \| `"low"`. `"high"` = must happen today, time-sensitive or deal-critical. `"medium"` = important but flexible timing. `"low"` = do when possible. |
| `deadline` | string \| null | Deadline expressed as a time string (`"before 15:45"`), relative phrase (`"today"`, `"this week"`), or ISO 8601. Null if open-ended. |
| `source_type` | enum | Where this action originated: `"email"` \| `"slack"` \| `"calendar"` \| `"intelligence"` \| `"manual"`. |
| `source_ref` | string \| null | The `id` of the source item in the corresponding section (e.g. `"email_act_1"`). Null for manually added items or items synthesized from multiple sources. For deduplication, always use `fingerprint` — not this field. |
| `url` | string \| null | Direct link to the resource that needs acting on — Gmail thread, Slack message, Google Doc, etc. |
| `deal_tag` | string \| null | Deal id if this action belongs to a deal. |
| `initiative` | string \| null | Initiative `id` from `briefing_config.yaml initiatives[]`. Same matching rules and consistency requirements as on email/calendar items — assigned in Phase 3.5 Step B. |
| `done` | boolean | `false` at creation. Set to `true` by the display component when the user checks it off. **Never reset to `false` by a generation pass** — match by `fingerprint` and carry forward. |
| `created_at` | ISO 8601 datetime | Timestamp of the run that first generated this action. Set once on creation; never updated. |
| `completed_at` | ISO 8601 datetime \| null | Timestamp when `done` was set to `true`. Set by the display component, or by the skill if it detects the underlying source has been resolved (e.g. email replied to, calendar RSVP accepted). `null` while open. |

### Fingerprint construction rules

The fingerprint is a short, human-readable string that uniquely identifies the underlying real-world action. It must be derivable from the source data alone — no positional information.

| `source_type` | Fingerprint format | Example |
|---|---|---|
| `email` | `"email-{gmailMessageId}"` | `"email-19dbbcf416d50d7c"` |
| `slack` | `"slack-{channelId}-{ts_nodot}"` (remove `.` from Slack ts) | `"slack-C0AQHD40GTH-17461234567890"` |
| `calendar` | `"calendar-{googleEventId}"`, or `"calendar-{slug}-{startISOMinute}"` when no event id is available (start time gives stable identity for recurrence-spawned events without ids) | `"calendar-abc123xyz_20260501"`, `"calendar-haize-labs-2026-05-04T20:30"` |
| `manual` | `"manual-{normalized-slug}"` (first 48 chars of text, lowercase, spaces → `-`, strip punctuation) | `"manual-review-puppygraph-roster-before-gabriel"` |
| `intelligence` | `"intel-{normalized-slug}"` (same slug rule as manual, from `text`) | `"intel-dynatrace-acquires-rookout-series-b"` |

The validator's pattern accepts the union of all the above, including `:` in the timestamp variant.

For `source_type: "email"`, the Gmail message ID is the string after `#all/` in the URL — always extract it from the URL at action-generation time, not from the positional `source_ref`.

If the same underlying source generates an action across two consecutive runs, the fingerprint will be identical and the merge algorithm will treat them as the same item.

### Merge algorithm (applies on every run type)

```
base_index = { item.fingerprint: item for item in base.action_items }
output = []

for each newly_derived_item:
    existing = base_index.get(newly_derived_item.fingerprint)
    if existing:
        # Same action seen before — update mutable fields, preserve completion state
        newly_derived_item.done         = existing.done
        newly_derived_item.created_at   = existing.created_at
        newly_derived_item.completed_at = existing.completed_at
        newly_derived_item.id           = existing.id   # keep stable display id
    else:
        # New action — set creation timestamp, leave done=false
        newly_derived_item.created_at   = now
        newly_derived_item.done         = false
        newly_derived_item.completed_at = null
    output.append(newly_derived_item)

# Preserve done-but-not-in-new-list items (source may have scrolled out of window)
for each existing_item in base.action_items:
    if existing_item.done == true:
        if existing_item.fingerprint NOT in {i.fingerprint for i in output}:
            output.append(existing_item)   # keep completed items even if source gone

sort output by priority desc, deadline asc
```

This algorithm runs on **every run type** — full, midday, and incremental — because all three re-derive the action list and all three must preserve completion state.

**Ordering:** Sort by `priority` descending, then by `deadline` ascending (earliest deadlines first within the same priority tier). Done items sort to the bottom regardless of priority.

**Rendering:** Render as a checklist. Each item gets a checkbox (`done` toggle), priority badge (color-coded), and the `text`. Done items render with strikethrough and move to a collapsed "Completed" section. If `url` is set, link the action text. If `deadline` is set, show it as a small badge. If `deal_tag` is set, show the deal badge. If `completed_at` is set, show the completion time next to the strikethrough text.

## Section: `focus_prompt`

A 2–4 sentence strategic framing for the day. Written by the generation pass. It synthesizes the day's most important threads — the active deals, the key decision points, the tone — into a single orienting paragraph. It should be opinionated: it tells Stefan what *actually* matters today, not a neutral summary.

```json
"focus_prompt": "It's Week 3 of Aspen diligence and Oli is paying close attention — two replies before 3am this morning says everything you need to know. Between the gitstat follow-up, the merger agreement security holdback, and tonight's exec update with Oli on Bits + Culminate, today is a day to move fast on the detail work so the big room conversations land well. Your morning is free — use it."
```

**Rendering:** Render as a styled blockquote or highlighted card — visually distinct from the rest of the briefing. No icon. The tone is direct and personal.

A fun fact and a joke to close the briefing. Both should be contextually relevant — tied to the day's meetings, deals, or intelligence items when possible.

## Source Link Policy

Every item surfaced in the briefing **must** carry a direct, clickable URL to its source. This applies to email items, Slack messages, calendar events, Google Drive documents, and web intelligence items. The generation pass must capture URLs during data fetching — not reconstruct them afterward.

### Required links by source type

| Source | Field | Link format |
|---|---|---|
| Email (act_today / fyi) | `url` | `https://mail.google.com/mail/u/0/#all/{messageId}` — use the message `id` from the Gmail MCP response |
| Slack channel | `url` on `slack.channels[]` | `https://dd.slack.com/archives/{channelId}` |
| Slack message | `url` in `messages[].links[]` | `https://dd.slack.com/archives/{channelId}/p{ts_no_dot}` (remove the `.` from the timestamp) |
| Calendar event | `links.other` | `event.htmlLink` from the Calendar MCP response |
| Calendar video | `links.zoom` | `event.hangoutLink` or the conferenceData entry URL |
| Google Doc / Sheet / Slide | extracted into `links[]` arrays | Full `https://docs.google.com/...` URL — extract from email body, Slack message, or calendar description at read time |
| Google Drive file | any `links[]` field | `file.webViewLink` from Drive MCP response |
| Intelligence item | `intelligence[].items[].url` | Article/post URL from WebSearch result |
| Action item | `action_items[].url` | Direct link to the resource needing action (Gmail thread, Slack message, Google Doc) |
| Meeting prep file | `meeting_preps[].file` | `computer://` path resolved by the display component |

### Null policy

`url: null` is allowed **only** when the source type definitively has no web URL (e.g. an in-person conversation, a phone call, an automated internal system with no deep-link). In that case, add a sibling `url_note` field explaining why:

```json
{
  "url": null,
  "url_note": "Internal Workday system — no direct deep-link available"
}
```

Do not omit the `url` field entirely. Null + url_note is the correct form. Prefer fetching a missing URL over emitting `url: null` if the URL is obtainable.

### Capture at fetch time

Google Doc links, Slack channel IDs, and Gmail message IDs are easiest to capture when the data is first read. Scan email bodies, Slack message text, and calendar descriptions for embedded URLs during Waves 1–3. Do not defer URL extraction to the JSON assembly phase.

## Intraday Update Behavior

When the generation system runs a mid-day or evening update (update_sequence ≥ 2):

- `meta.generated_at` and `meta.update_sequence` are updated.
- `meta.next_meeting` is recomputed from current time.
- `calendar.events` are updated if the calendar changed (new invites, cancellations).
- `calendar.conflicts` are regenerated.
- `email` is refreshed with new arrivals since last pass.
- `slack` is refreshed with messages since last pass (update `since` to reflect the tighter window).
- `intelligence` is typically unchanged for intraday passes unless a major item breaks.
- `action_items` are regenerated — items marked `done: true` in the previous version are preserved as `done: true`. New items are appended. Completed items should not be removed (the display component can filter them).
- `focus_prompt` and `fun` are refreshed if significantly new context warrants it; otherwise kept from the morning pass.

## File Locations

```
src/
  daily/
    briefing_schema.md      ← this file (schema + rendering spec)
    briefing_config.yaml    ← static configuration (deals, channels, categories)

briefings/
  daily/
    daily.json              ← the live briefing, overwritten on each update pass
  2026_04_27/
    meeting-prep-1545-adaptive-ml.md
    ...
```

The generation pass always writes to `briefings/daily/daily.json`, overwriting the previous version. The `meta.update_sequence` and `meta.generated_at` fields inside the JSON record which pass produced it. The display component always reads `briefings/daily/daily.json` — no version resolution needed.
